# Transcript

**长文本语音转文字整理为结构化讲稿——解决大模型处理长文本的 "Lost in the Middle" 问题。**

直接扔给大模型处理 3 万字以上的语音转写，模型注意力随文本长度衰减，中间段落的关键信息——模型名称、价格数据、企业案例——默默消失。Transcript 将全文按话题切段，每段由独立 Agent 处理，确保任意位置的术语、数据、金句完整保留。

## 效果对比

同一份 3.8 万字访谈转写，全量直出 vs 本 Skill 分段处理：

| | 全量直出 | 本 Skill |
|------|------------|-----|
| 章节标题 | **0**，一个长段落流 | **18 个**，可精准跳转 |
| 总字数 | 13,500 | 21,345 (**+58%**) |
| 定量数据点 | 44 | **67 (+52%)** |
| 核心模型名称 | **0 次**（完全丢失） | 15 次完整保留 |
| 关键设备供应商 | **0 次**（完全丢失） | 2 次完整保留 |
| Token 优先权案例 | **0 次**（完全丢失） | 2 次完整保留 |
| 国产芯片型号 | **0 次**（完全丢失） | 2 次完整保留 |
| 全文核心概念 | 31 次 | **56 次 (+80%)** |
| 企业反制细节 | 6 次 | **9 次 (+50%)** |

## 设计原理

```
原始转写文本
  → 扫描 Agent（Haiku，通读全文找话题边界）
  → 校验 Agent（Haiku，验证话题与内容匹配）
  → 按话题切段（±2 行自然重叠）
  → N 个 Agent 全量并行处理（每段独立上下文）
  → 完整性检查 + 文件有效性验证
  → 拼接整理稿 + 质检 Agent + 速览 Agent 并行
```

- **反 Lost in the Middle**：每段 Agent 仅处理自己片段的短上下文，不存在注意力衰减
- **全量并行**：所有片段同时处理，总耗时 = 最慢的一个，不是累加
- **保守去噪**：去除口头禅和填充词，但术语、数据、专有名词原样保留，拿不准绝不替换
- **自动质检**：拼接后采样检查跨段重复、ASR 残留、术语一致性

## 产出

| 产出 | 说明 |
|------|------|
| **整理稿** | 去噪 + 段落精炼 + 章节标题，信息零遗漏，可流畅阅读 |
| **速览稿**（可选） | 结构化要点速查，表格/列表为主，5 分钟了解全貌 |
| **质检报告**（可选） | 跨段重复、ASR 错字、术语一致性检查结果 |

## 性能

以 3.8 万字文本（16 段）为例：

| 阶段 | 耗时 |
|------|------|
| 扫描 + 校验 | ~1.5 分钟 |
| 16 段全量并行整理 | ~1.5 分钟 |
| 质检 + 速览并行 | ~1.5 分钟 |
| **总计** | **~4.5 分钟** |

短文本（< 8,000 字）自动走快速通道，单 Agent 直接处理。

## 安装

所有 AI Coding 平台共用同一份 `SKILL.md` + `scripts/`，复制到对应目录即可：

```bash
# Claude Code
cp -r . ~/.claude/skills/transcript.skill/

# Gemini CLI
cp -r . ~/.gemini/skills/transcript.skill/

# Cursor
cp -r . ~/.cursor/skills/transcript.skill/

# Codex
cp -r . ~/.codex/skills/transcript.skill/

# OpenCode
cp -r . ~/.config/opencode/skills/transcript.skill/
```

依赖 Python 3（仅用于 `text_stats.py` 规模判定）。

## 使用

对话中传入文件路径：

```
/transcript ~/Downloads/会议录音转文字.txt
```

或自然语言触发：

> 帮我整理这份访谈记录

默认产出整理稿。回复「速览」追加速览稿。
